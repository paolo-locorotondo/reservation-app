## Step rimanenti

### 1. Smoke test locale immagini (consigliato, 5 min)

```powershell
# Avvio (build già fatta)
docker compose -f docker-compose.test.yml up

# In altro terminale, smoke API
curl http://localhost:3001/api/sites

# Browser → http://localhost:3000
# - login Google (deve già funzionare con redirect URI dev configurato)
# - prenota un posto
# - cancella la prenotazione
# - verifica "Le mie prenotazioni"

# Stop
docker compose -f docker-compose.test.yml down
```

Variabili `.env` necessarie: `DATABASE_URL`, `DIRECT_URL`, `NEXTAUTH_SECRET`, `NEXTAUTH_URL=http://localhost:3000`, `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`.

### 2. Setup IBM Cloud CLI

```powershell
# Install (Windows)
iex (New-Object Net.WebClient).DownloadString('https://clis.cloud.ibm.com/install/powershell')

# Plugin necessari
ibmcloud plugin install code-engine
ibmcloud plugin install container-registry

# Login
ibmcloud login --sso
ibmcloud target -r eu-de   # o eu-gb / us-south, da decidere
ibmcloud target -g <resource-group>   # tipicamente "Default"
```

### 3. Container Registry

```powershell
# Login al registry regionale
ibmcloud cr region-set eu-de   # match con la region scelta sopra
ibmcloud cr login

# Crea namespace (deve essere globalmente unico)
ibmcloud cr namespace-add reservation-app
```

### 4. Tag + push immagini

```powershell
# Tag
docker tag reservation-app/api:local de.icr.io/reservation-app/api:v1
docker tag reservation-app/web:local de.icr.io/reservation-app/web:v1

# Push
docker push de.icr.io/reservation-app/api:v1
docker push de.icr.io/reservation-app/web:v1
```

### 5. Code Engine project

```powershell
ibmcloud ce project create --name reservation-mvp
ibmcloud ce project select --name reservation-mvp
```

### 6. Deploy app `api` (visibility internal)

```powershell
ibmcloud ce app create `
  --name api `
  --image de.icr.io/reservation-app/api:v1 `
  --registry-secret <auto-generated> `
  --port 3001 `
  --visibility project `
  --min-scale 0 `
  --max-scale 3 `
  --memory 0.5G --cpu 0.25 `
  --env DATABASE_URL=<supabase-pooler-url> `
  --env DIRECT_URL=<supabase-direct-url> `
  --env NEXTAUTH_SECRET=<same-as-web> `
  --env ADMIN_EMAILS=<csv>
```

Per la registry-secret la prima volta serve generarla con `ibmcloud ce registry create`.

### 7. Deploy app `web` (visibility public)

```powershell
ibmcloud ce app create `
  --name web `
  --image de.icr.io/reservation-app/web:v1 `
  --registry-secret <same-as-api> `
  --port 3000 `
  --visibility public `
  --min-scale 0 `
  --max-scale 3 `
  --memory 0.5G --cpu 0.25 `
  --env API_INTERNAL_URL=http://api.<project-internal-domain> `
  --env NEXTAUTH_SECRET=<same-as-api> `
  --env NEXTAUTH_URL=https://<web-public-url> `
  --env GOOGLE_CLIENT_ID=<...> `
  --env GOOGLE_CLIENT_SECRET=<...>
```

L'URL interno preciso si scopre con `ibmcloud ce app get --name api`. Tipicamente è `api.<project-id>.<region>.codeengine.appdomain.cloud` con visibility internal-only.

L'URL pubblico del web si scopre con `ibmcloud ce app get --name web` — quel valore va nel `NEXTAUTH_URL` (chicken-and-egg: prima crea web con un placeholder, poi update con l'URL reale via `ibmcloud ce app update`).

### 8. Aggiornare Google OAuth

Su [Google Cloud Console → Credentials](https://console.cloud.google.com/apis/credentials) aggiungere:

```
https://<web-public-url>/api/auth/callback/google
```

agli **Authorized redirect URIs** dell'OAuth client esistente.

### 9. Verifica produzione

- Apri l'URL pubblico del web
- Login con Google → deve redirezionare al callback prod
- Prenota / cancella / "Le mie prenotazioni"
- `ibmcloud ce app logs --name api --follow` se qualcosa non torna

## Costo atteso

- Code Engine free tier (100K vCPU-s + 200K GB-s + 100K req/mese): **£0** finché traffico MVP.
- Container Registry free (500MB storage): **£0**.
- Supabase free (500MB DB + 50K MAU): **£0**, con pausa progetto dopo 7gg di inattività (riavviabile in un click).

**Totale: £0/mese**.

## Note / decisioni prese

- DB managed IBM Cloud (`Databases for PostgreSQL`) **scartato** perché senza free tier (~£60-80/mese minimo).
- OpenShift / IKS scartati a favore di Code Engine per ridurre il setup (no cluster da gestire, scale-to-zero nativo).
- `pnpm deploy --legacy` invece di `inject-workspace-packages=true` nel workspace, per non toccare la config di workspace e impattare il dev locale.
- Dockerfile usano `node:20-bookworm-slim` (Debian, glibc) e non Alpine (musl) per compatibilità con i binari Prisma `debian-openssl-3.0.x`.
